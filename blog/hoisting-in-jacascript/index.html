Hoisting in javascript: what it is and how it works

If you've ever seen a JavaScript variable or function used before it's declared and somehow still work, you've already met hoisting.

Hoisting can be confusing at first, but once you understand the rules, it becomes a powerful concept that helps explain a lot of JavaScript‚Äôs behavior.

üß† What is hoisting?

Hoisting is JavaScript's default behavior of moving declarations to the top of their scope before the code is executed.

In simpler terms:

JavaScript reads through your code first and "lifts" variable and function declarations to the top of their scope.

This doesn't mean your code literally moves‚Äîjust that the engine _interprets_ it as if the declarations were at the top.

Here‚Äôs a simple example:

Why doesn‚Äôt this throw an error?

Because JavaScript sees it like this:

console.log(a); // undefined
a = 5;

The declaration (var a) is hoisted, but the assignment (= 5) is not.

‚ùå Let and const are different

With let and const, declarations are also hoisted‚Äîbut they're not initialized.

console.log(b); // ReferenceError

This happens because of the Temporal Dead Zone‚Äîthe time between the start of the block and the actual declaration. During this period, the variable exists but can‚Äôt be accessed.

So:

  • var is hoisted and initialized with undefined
  • let and const are hoisted but not initialized

üß© Function hoisting

Function declarations are fully hoisted‚Äîincluding their body.

greet(); // "Hello!"

function greet() {
  console.log("Hello!");
}

JavaScript moves the entire function definition to the top of the scope.

But be careful:

Function expressions are not hoisted the same way:

var sayHi = function () {
  console.log("Hi!");
};

In this case, sayHi is hoisted as a var, but it‚Äôs initialized with undefined. So when you call it, it's not a function yet.

üìå Conclusions

Hoisting can feel like magic, but it's just how JavaScript parses and prepares your code before running it.

Understanding hoisting helps you avoid bugs like using a variable before it's ready, or accidentally calling a function expression too soon.

If you ever wonder why something is undefined or throwing a ReferenceError, check if hoisting is the reason.
