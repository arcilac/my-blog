CAP Theorem

Understanding the Trilogy of Consistency, Availability, and Partition Tolerance

Imagine you're designing a distributed system, like a database that needs to be available in different locations (servers) at the same time. This is where the CAP theorem (also known as Brewer’s theorem) comes into play. It is a fundamental principle in distributed system design that states that in the presence of a network partition (communication failures between nodes), you can only guarantee two out of these three properties:

  1. Consistency (C): All nodes see the same data at the same time.
  2. Availability (A): Every request receives a response, regardless of whether some nodes have failed.
  3. Partition Tolerance (P): The system continues to operate even if there are communication failures between nodes.

¿Cómo usar el teorema CAP en la vida real?How to Use the CAP Theorem in Real Life?

The CAP theorem doesn’t mean you have to choose only two and completely forget about the third forever. Instead, it serves as a framework for making decisions. When designing a system, you must prioritize what is most important for your use case.

For example:

  • If you need precise banking transactions, consistency is key (CP: Consistency + Partition Tolerance).
  • If you're designing a social media service, where a quick response is always required, availability is the priority (AP: Availability + Partition Tolerance).
  • In some rare cases, when the network is stable and communication can be guaranteed, you can focus on consistency and availability (CA), though this is uncommon in real-world distributed systems.

Which One to Choose?

Choosing the right model depends on the needs of your application. Here are some examples of databases that prioritize different aspects of the CAP theorem:

  • CP (Consistency + Partition Tolerance): MongoDB, HBase
  • AP (Availability + Partition Tolerance): Cassandra, DynamoDB
  • CA (Consistency + Availability): PostgreSQL, MySQL (when not distributed)

The key is to evaluate what is most critical for your application and make smart trade-offs.
